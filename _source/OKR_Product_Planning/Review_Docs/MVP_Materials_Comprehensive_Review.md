# MVP Strategy & Execution Review – October 2025

## Strategic Alignment
- **High – Launch target drift**: The strategy still promises a November 30 launch (Karvia_OKR_Product_Planning/01_MVP/MVP_STRATEGY.md:4) while the execution README has quietly slipped to December 17 (Karvia_OKR_Product_Planning/01_MVP/README.md:7). Call out the 2½‑week slip in stakeholder comms and re-baseline downstream milestones that assume an 8-week window.
- **High – Tiered feature scope vs plan**: The MVP strategy lists six iBrain modules as “always-on toggles” for the MVP (Karvia_OKR_Product_Planning/01_MVP/MVP_STRATEGY.md:23-214), yet the Week 5-12 schedule only budgets time for core screens and never assigns work for predictive analytics, sentiment, coaching, or toggle UX (Karvia_OKR_Product_Planning/MASTER_DEV_LIST.md:257-335). Decide whether these belong in MVP or defer them formally before sprint capacity gets consumed by undocumented scope.
- **Medium – Engine inventory mismatch**: Planning documents freeze the architecture at six engines (Karvia_OKR_Product_Planning/01_MVP/MVP_STRATEGY.md:160-167), but the repo boot scripts already include eight (`dev:insights`, `dev:collaboration`) without matching setup tasks (package.json:12-20, package.json:39-41). Update the roadmap to cover these engines or archive them to avoid wasted install/build cycles.

## Design Review
- **High – Team screen parity gap**: The Week 5 plan only covers CRUD and role visibility, but the finalized mockup expects workload, engagement, recognition nudges, and filter chips (Karvia_OKR_Mockups/Finalised_Mockups/05_team.html:208-316). Without backend metrics, the shipped screen will feel unfinished; either down-scope the UI or schedule APIs for capacity, engagement, and action cards.
- **High – Planning screen complexity**: The planning mockup relies on AI reasoning blurbs, per-person approval workflows, estimated effort, and context chains (Karvia_OKR_Mockups/Finalised_Mockups/06_planning.html:200-312). No week currently assigns data modeling or services to feed those elements, so start carving out API contracts before Week 9 or replace the mockup with an achievable interim design.
- **Medium – Objectives page duplication**: Day 4 instructions tell the team to copy the mockup into `client/pages/objectives.html` (Karvia_OKR_Product_Planning/Daily_Handoffs/Week_5/WEEK_5_PLAN.md:102-154) even though that page already exists with the same Tailwind scaffold (client/pages/objectives.html:1-40). Reuse the existing asset and focus the effort on wiring live data and filters.

- **High – Management workflow gaps**: `ADMIN-003 Manage Users` is flagged for Week 5 (Karvia_OKR_Product_Planning/01_MVP/MVP_USER_STORIES.md:674), but `server/routes/admin.js` is still a placeholder (server/routes/admin.js:1-11) and no tasks cover invitation/role management. Either move the story to a later week or schedule the missing API/UI work now.
- **High – AI OKR approval bug persists**: The client still expects `data.suggestion` (client/js/ai-okr-api-client.js:73-76) while the API returns `{ data: { suggestions: [...] } }` (server/routes/ai-okr.js:186-193), which explains the empty review page. - **Medium – Persona duplication**: Multiple Week 5 stories overlap (`EXEC-003` vs `EXEC-005`, `EMP-004` vs `MGR-008`), yet estimates treat them independently. Consider consolidating or at least splitting acceptance into shared components to prevent double-counting UI work.

## Architecture Review
- **High – Shared models still missing**: Engines continue to `require('../../server/models/*')` (engines/iam/index.js:109-110), contradicting the Week 0 mandate that everything go through `@karvia/shared-models` (Karvia_OKR_Product_Planning/01_MVP/MVP_STRATEGY.md:171-185). Until the package exists, true microservice isolation and external deployment remain impossible.
- **High – Referenced models don’t exist**: User, Objective, Goal, and Task schemas all reference `Team` and `Department` models (server/models/User.js:100-110; server/models/Objective.js:174-183), yet no `server/models/Team.js` or `Department.js` ships today. Plan the schema creation, migration, and population steps before Week 5 relies on these relations.
- **Medium – Admin & analytics toggles undefined**: The strategy requires per-feature toggles under `ibrain_config` (Karvia_OKR_Product_Planning/01_MVP/MVP_STRATEGY.md:193-214), but there is no flag loader or fallback path in the codebase. Introduce stubbed toggles now so optional engines (Redis/OpenAI) don’t hard-crash in demo environments.
