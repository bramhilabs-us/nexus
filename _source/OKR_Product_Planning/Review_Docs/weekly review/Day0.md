# Day 0 Review (Shared Middleware & Utilities)

- **Critical** – `authenticateToken` and `optionalAuth` call `GET /api/auth/verify`, but the IAM engine only exposes `POST /api/auth/validate`; every protected request will therefore return a 404/503 despite a valid token (`server/middleware/authGuards.js:36`, `server/middleware/authGuards.js:169`, `engines/iam/index.js:414`). Fix the endpoint/method or add the missing IAM route before wiring this middleware into routes.
- **Critical** – The IAM response does not provide the shape expected by the new guards: `authenticateToken` copies `verifyResponse.data.user` as-is, but that payload lacks `_id` and `managed_businesses`, so downstream middleware breaks (e.g., `generalAPILimiter` falls back to IP-throttling, and consultants are blocked because `managed_businesses` is undefined) (`server/middleware/authGuards.js:51`, `server/middleware/rateLimiting.js:118`, `server/middleware/roleGuards.js:145`, `engines/iam/index.js:437`). Either extend the IAM response or massage the object before attaching it to `req.user`.
- **High** – `authenticateTokenLocal` uses a different default secret (`karvia-dev-secret-key-2024-change-in-production`) and expects `decoded.email`, so locally verified tokens minted by the IAM engine (`karvia-business-secret`, no email in payload) will always fail (`server/middleware/authGuards.js:108`, `server/middleware/authGuards.js:113`, `engines/iam/index.js:108`). Align the secret and payload extraction with the IAM token format.
- **High** – `requireConsultantAccess` assumes `req.user.managed_businesses` contains raw business IDs, but the current auth response never populates that field; consultants will be denied access even when authorised (`server/middleware/roleGuards.js:144`, `engines/iam/index.js:437`). Ensure the token verification step injects `managed_businesses` (and normalise ObjectId/string comparisons) before enforcing this guard.
- **Medium** – The new `requireManagerAccess` middleware is a stub that merely checks role and then `next()` without verifying ownership (`server/middleware/roleGuards.js:208`). Either finish the relation check (e.g., ensure `target_user.manager_id === req.user._id`) or remove the helper to avoid a false sense of security.
