System Architecture Overview

3-tier architecture: Client → API → Database

┌──────────────────────────────────────────────────────────────┐ │ CLIENT LAYER (Vanilla JS + Bootstrap) │ │ • Dynamic pages (dashboard, objectives, analytics) │ │ • API client wrappers (fetch + error handling) │ │ • Client-side calculators (progress %, status) │ │ • Responsive design (desktop + mobile) │ └──────────────────────────────────────────────────────────────┘ ↓ HTTPS ┌──────────────────────────────────────────────────────────────┐ │ API LAYER (Node.js + Express) │ │ • RESTful routes (13 modules, 50+ endpoints) │ │ • Middleware (auth, RBAC, validation, error handling) │ │ • Service layer (business logic, external APIs) │ │ • iBrain integration (toggle-controlled) │ └──────────────────────────────────────────────────────────────┘ ↓ SQL ┌──────────────────────────────────────────────────────────────┐ │ DATA LAYER (PostgreSQL + Redis) │ │ • PostgreSQL: Core data (users, businesses, objectives) │ │ • Sequelize ORM: Models + migrations │ │ • Redis: Session cache, rate limiting (Beta) │ │ • Business-level data isolation (multi-tenant) │ └──────────────────────────────────────────────────────────────┘
Design Principles: Stateless API servers (horizontal scaling), business-level data isolation (multi-tenant security), zero hardcoded values (config-driven), graceful degradation (iBrain toggle).

Database Schema

10 core models, business-level data isolation

Business

Purpose: Multi-tenant root (business_id isolation)

Fields: name, ibrain_enabled, ibrain_api_key, settings

Relationships: hasMany Users, Assessments, Objectives

User

Purpose: Authentication + RBAC

Fields: email, password_hash, role, business_id

Roles: admin, manager, team_lead, member

AssessmentTemplate

Purpose: Template library (org/team/individual)

Fields: name, level, category, business_id

Relationships: hasMany AssessmentQuestions

AssessmentQuestion

Purpose: Questions for assessments

Fields: text, dimension (Speed/Strength/Intelligence), category

Scoring: 1-5 Likert scale

Assessment

Purpose: Completed assessments with SSI scores

Fields: template_id, scores (JSON), insights, business_id

Calculations: SSI scores, gap analysis

Objective

Purpose: OKRs (multi-level cascade)

Fields: title, level, parent_id, key_results (JSON), business_id

Progress: Auto-calculated from KR updates

AIOKRSuggestion

Purpose: iBrain-generated objectives (pre-approval)

Fields: assessment_id, objective_text, status, business_id

Workflow: Generated → Reviewed → Approved → Saved to Objective

Invitation

Purpose: Team member invitations

Fields: email, role, token, status, business_id

Workflow: Sent → Pending → Accepted

Goal (Legacy)

Purpose: Original goal tracking (pre-OKR)

Status: Deprecated (use Objective instead)

Migration: V1 (migrate to Objective model)

Task (Legacy)

Purpose: Task management (pre-OKR)

Status: Deprecated (use Objective KRs instead)

Migration: V1 (migrate to KRs)

Data Isolation: All queries scoped by business_id (WHERE business_id = :business_id). Multi-tenant security enforced at ORM level (Sequelize scopes).

Data Model Relationships

How data flows through KARVIA Pro

Business (1) ────┬──────> User (N) │ ├──────> AssessmentTemplate (N) │ │ │ └──────> AssessmentQuestion (N) │ ├──────> Assessment (N) │ │ │ └──────> AIOKRSuggestion (N) │ │ │ └──────> Objective (N) │ ├──────> Objective (N) │ │ │ └──────> Objective (N) [parent_id self-reference] │ └──────> Invitation (N) **Key Relationships**: 1. Business → User (1:N): One business has many users 2. Business → Assessment (1:N): One business has many assessments 3. Assessment → AIOKRSuggestion (1:N): One assessment generates many AI suggestions 4. AIOKRSuggestion → Objective (1:1): Approved suggestion becomes objective 5. Objective → Objective (1:N): Parent objective has child objectives (cascade) 6. Business → Objective (1:N): One business has many objectives (all levels) 7. AssessmentTemplate → AssessmentQuestion (1:N): One template has many questions
Cascade Model: Objectives support multi-level hierarchy via parent_id (Company → Department → Team → Individual). Progress rolls up from child to parent.

Authentication & Authorization

JWT authentication + RBAC + business-level isolation

Authentication Flow

  • Registration: POST /api/auth/register → Create user + business
  • Login: POST /api/auth/login → Validate credentials → Issue JWT (15min expiry)
  • Token Format: { userId, businessId, role, email, exp }
  • Refresh: Refresh tokens (V1 feature, not MVP)
  • Logout: POST /api/auth/logout → Invalidate token (client-side)

Security

  • Password Hashing: bcrypt (10 salt rounds)
  • JWT Secret: 256-bit key (rotated quarterly)
  • HTTPS Only: Secure cookie (httpOnly, secure flags)
  • Rate Limiting: 5 failed logins → 15min lockout (Beta)

Role-Based Access Control (RBAC)

  • admin: Full access (manage business, users, all OKRs)
  • manager: Department-level (create/edit team OKRs, view analytics)
  • team_lead: Team-level (create/edit individual OKRs, view team analytics)
  • member: Individual-level (update own KRs, view company OKRs)

Business-Level Isolation

  • JWT includes business_id: All queries scoped by business
  • ORM Scopes: Sequelize default scope (WHERE business_id = :id)
  • Multi-Tenant Security: Business A cannot access Business B data
  • Audit Logging: All CRUD operations logged with business_id (V1)
Middleware Stack: authenticate() → checkBusinessAccess() → checkRole() → validate() → route handler → errorHandler().

Frontend Architecture

Vanilla JS + Bootstrap, no framework bloat

Technology Choices

  • No Framework: Vanilla JS (ES6+), no React/Vue/Angular
  • Reasoning: Faster page loads, easier handover, no build step
  • UI Library: Bootstrap v5.3 (responsive grid, components)
  • Icons: Bootstrap Icons (no Font Awesome dependency)
  • Charts: Chart.js (analytics dashboards, Week 3)

Key Pages

  • dashboard.html: Company overview, top 3 objectives
  • objective-detail.html: Single objective deep dive (Week 4)
  • analytics.html: Team analytics, drill-down (Week 3)
  • assessment.html: Assessment form, SSI scoring (Week 1)

Client-Side Architecture

  • API Client Wrappers: Fetch + error handling (client/js/api-client.js)
  • Calculator: Client-side progress calc (mirrors server, Week 4)
  • State Management: localStorage (JWT token, user prefs)
  • Routing: Multi-page app (no SPA, faster initial load)

Performance

  • Page Load: <2s (dashboard), <1s (list views)
  • API Calls: <500ms p95 (real-time updates)
  • Bundle Size: <100KB (no webpack, just vanilla JS)
  • Mobile: Responsive (Bootstrap grid, tested iOS/Android)
Why No Framework? Faster page loads (no 500KB React bundle), easier handover (any dev knows vanilla JS), no build tooling (simpler deployment).
