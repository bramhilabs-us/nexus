# Consultant Beta Operations

<!-- @GENOME T2-PRD-015 | ACTIVE | 2026-03-30 | parent:T2-PRD-011 | auto:/coding | linked:- -->

## 1. Objective

Make the beta repeatable across companies.

The consultant is a core part of the beta operating model, not an optional layer around the software.

## 2. Weekly Beta Cadence

### Monday

- review objective status
- review blocked and deferred tasks
- identify where work is returning to the same owner
- set 1-3 priority intervention areas

### Mid-week

- review next moves with team leads
- adjust sequencing or ownership if needed
- capture notable handoff failures or wins

### End of week

- complete weekly evidence review
- capture interventions
- update next week's focus

## 3. Consultant Responsibilities

- interpret assessment signals
- translate them into objective focus
- help shape teams or pods
- guide how next moves are framed
- review blocked/deferred patterns
- capture weekly evidence

## 4. Product Support Required

The consultant must be able to see:

- company overview
- objective progress
- weekly goals
- blocked and deferred tasks
- evidence/review summary

## 5. What Consultants Should Not Need

Before beta, the consultant should not need:

- a brand-new portal
- a separate workflow tool
- a parallel reporting spreadsheet for core product data

If they need those things, the product changes are incomplete.

## 6. Beta Review Template

Each weekly review should answer:

1. What moved this week?
2. What got blocked?
3. Where did work return to the same person?
4. Which team or objective needs intervention?
5. What is the next most important move?

## 7. Success Criteria

- every beta company follows the same weekly cadence
- intervention logging is consistent
- consultant observations can later be compared across companies
